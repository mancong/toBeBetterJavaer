# 为什么 LLM 没有记忆？

面试官问你："为什么 LLM 没有记忆？"如果你回答"上下文窗口不就是记忆吗？现在都百万 token 了，记忆够用了。"这道送分题你又没把握住，呜呜呜。

为什么？

因为上下文窗口是 RAM，不是硬盘。会话一结束，全没了。

哈喽大家好，我是二哥呀。今天用 3 分钟，带你搞懂为什么 LLM 没有记忆。

面试官问这道题，考三层：第一，LLM 为什么在架构层面就没有记忆；第二，上下文窗口和真正的记忆有什么区别；第三，Agent 时代怎么给 LLM 补上记忆。

好，接下来给你满分回答，照着背就完事了。

LLM 就像一个每天醒来就失忆的天才医生。医术精湛，但完全不记得昨天的病人。护士不把病历念一遍，医生就把老病人当新病人。

为什么会这样？因为 Transformer 是无状态的纯函数。训练阶段学到的知识压进了权重，但推理时权重是冻结的——没有反向传播、没有梯度更新、没有学习发生。你和它聊了一小时，它的权重一个数字都没变过。每次调用，完全独立。

那上下文窗口呢？窗口是工作台上的临时便签。医生看诊时，护士把这次对话的记录都摊在桌上，医生可以翻看。但门诊结束，便签全部扔掉。而且窗口还有个 Lost in the Middle 效应——开头和结尾的信息记得住，中间的容易丢。

能不能靠训练让它"记住"？三重问题。第一，灾难性遗忘（Catastrophic Forgetting）——学了新的，忘了旧的。第二，不实时——用户说"我换工作了"，你不可能为这句话重新训练一次模型。第三，成本高——一次微调的算力开销远超一次推理。

那怎么给 Agent 补上记忆？三层架构。

短期记忆，就是对话历史，存在上下文窗口里，会话内有效。

长期记忆，外部持久化存储。Claude Code 的 CLAUDE.md 和 memory 文件就是典型——跨会话存在，每次启动自动加载进上下文。

工作记忆（Working Memory），当前任务执行中产生的中间状态。比如 Agent 调了三个工具，前两个的返回结果就是工作记忆，第三个工具的调用要参考前面的结果。

面试官如果继续追问："既然没有记忆，为什么 ChatGPT 和 Claude 看起来有记忆？"

告诉他——那不是模型的记忆，是产品工程。ChatGPT 的 Memory 功能是一个外部系统，对话时自动提取关键信息存到数据库，下次会话开始前把相关记忆检索出来塞进上下文。Claude Code 的 memory 文件也是同样的思路——持久化到文件系统，启动时加载。Agent 看起来有状态，但底层每次调用仍然是无状态的 Transformer。有状态是基础设施的属性，不是模型本身的。

最后一句口诀——上下文是 RAM，关机就白搭；记忆靠外挂，Agent 才记得住你是谁。
