# AI 技术文章的生成工作流

## 环境声明

执行前跑 `date "+%Y年%m月%d日"` 拿当前日期。联网搜索关键词、frontmatter 的 date、正文时间描述，都用这个日期。

---

## 写作原则

### 语气和称呼

用"大家/我们/小伙伴"和读者拉近关系，少用"你"。

### 表达直给（全文强制）

一句话能说清楚的事，不要用两句话绕。先给结论，再补原因或细节。读者来看技术文章是为了拿到信息，不是听你做铺垫。

禁止的绕弯模式：先铺垫再说正事（"在介绍 X 之前，先了解一下 Y"）、假设式开场（"如果你遇到过 X，那么一定知道 Y"）、定义式起手（"所谓 X，就是指 Y"）、层层递进废话（"要理解 X，需要先明白 Y"）、同义反复（同一个判断换说法再说一遍）、无信息量的过渡（"说完了 A，接下来看看 B"）。

自检方法：逐段读，每句话问"删掉这句，读者会少知道什么？"——答案是"什么都不少"的句子就是绕弯，删。

### 双轨表达（全文强制）

正文的每一句话属于两轨之一，先判断轨道，再决定表达方式。判断分两步，**第一步优先级最高**：

1. **这句话和技术强相关吗？** 只要句子承载技术信息（机制、原理、参数、数据、操作步骤、技术结论），无论句子里有没有"我"，技术内容一律走技术论述轨。"我把 chunk 调到 512 就不卡了"——有"我"，但"chunk 调到 512"是技术信息，这部分必须准确书面表达
2. 不承载任何技术信息的句子，再用"去我测试"分轨：去掉"我"就没意义的（纯动作、遭遇、感受）走体验叙事轨

**技术论述轨（科学表达）**——所有技术强相关的句子：

- **准确无误，清清楚楚**：用词准确无歧义，读者看一遍就懂，不需要停下来猜。禁止用成语、网络梗、模糊比喻、生僻词描述技术行为——读者看完觉得莫名其妙的词，一律换成直白的书面表达
- **不省略**：主语、宾语、限定条件写全，不指望读者靠语境脑补。"压缩策略优先保留前缀部分，因为命中前缀缓存的 token 成本只有未命中的十分之一"——原因、对象、数据都在句子里
- **量化优先**：能给数字不给程度词。"大约 1700 行代码"而不是"体量很大"；"限制为 250 个字符"而不是"有长度限制"
- **一句一事实**：技术陈述句不叠修辞，判断和事实分开写——先陈述源码/文档怎么写的，再单独一句给自己的判断
- **可验证性**：每个技术断言要么有出处（源码、文档、实测），要么明确标注"我的推断是"，不把推断写成事实
- 禁止语气词、网络腔、拟人

**体验叙事轨（二哥声口）**——不承载技术信息、去掉"我"就没意义的句子，陈述的是我的动作、遭遇、感受：

- 允许真实的口语声口："问了一嘴，原来是我之前配置过了，虽然我也不记得，但无所谓了，用就用呗""这个咱真的有一说一，不黑不吹""太用心了兄弟"
- 实况叙事是这一轨的主体：装了什么、卡在哪、第一反应是什么、花了多少钱，按时间顺序讲，带着当时的毛边
- 素材来源必须真实（来自用户提供或实测记录），编造的体验一眼假

**两轨的边界（三条红线，越线即改）**：

1. **体验句不得携带技术断言**。"这界面看着太爽了"可以；"这架构写得太爽了，所以性能高"不行——句子后半段是技术判断，必须切回科学表达轨单独写。一句话里两轨混合时，技术部分永远按技术论述轨的标准执行，口语只能落在纯体验的部分
2. **口语声口有密度上限**：一个章节最多 2 处，纯技术拆解章节可以为 0。声口是调味料，超量就成了朋友聊天体
3. **两轨共同禁区**：叠用句号（。。）、连续感叹号、"tm/尼玛"类脏话腔、生造缩写词（"深思考""技报"）、随意口语代替正式用词（"管输出格式"应为"负责输出格式"）——这些在任何一轨都不出现

情绪和个性表达的更高浓度形态（金句）另有管制，见「金句与情绪表达」。

### 开场三件套（强制）

高赞文章的开头有固定结构，按顺序执行：

1. **打招呼**：第一句固定是"大家好，我是二哥呀。"独立成段
2. **钩子**：热点事件、反差冲突或直击痛点的问题（"有没有想过？Claude Code 的代码搜得又快又准，到底是怎么实现的？"）
3. **调查投入声明**：一句话说清为这篇文章付出了什么——"我花了一早上时间，翻了翻 Boris Cherny 的播客、亚马逊的论文、Claude Code 源码，把这件事从头到尾捋了一遍"。声明的内容必须和步骤 1.5/2 实际做过的调查一致，不夸大

可选第 4 件：发车句（"系好安全带，我们粗粗发～"）或表态句（"先叠个甲，我是 Claude Code 的重度用户，这篇不会贬低谁来凸显谁"）。签名元素清单见 `human-tone.md`，一篇最多用 1-2 个，评测类和有争议话题优先用表态句。

### 正文结构

`## 01、标题` / `## 02、标题`。标题只写名称，不加冒号后缀解释，错误例子：`## 01、标题：解释`。三级标题 `### xxx`，不加分类前缀。

### 问题驱动结构（深度拆解类强制，其他类型鼓励）

好的技术文章不是"我知道什么就写什么"，而是"读者会问什么，我按什么顺序回答"。

**操作方法**：
1. 确定主题后，先列出读者最可能问的 5-8 个问题（从"这是什么"到"怎么用好"逐步深入）
2. 砍掉太基础或太偏的，保留 4-7 个形成问题链
3. 每个章节标题就是一个问题（或问题的答案），章节内容就是回答这个问题
4. 问题之间有递进关系：前一个问题的答案自然引出下一个问题

每个问题的回答结构：**先给结论（一句话）→ 展开论证（证据+解读）→ 收一句实操建议**。问题链示例见 `references/deep-analysis.md`「结构模板」。

### Case 创意

尽可能有趣，让读者眼前一亮。

涉及 Agent 可以和 PaiAgent 结合、RAG 可以和派聪明结合、CLI可以和 PaiCLI 结合（路径见步骤 1.5）。

### 段落与列表各得其所

正文默认用段落表达，用完整句子和自然过渡。能用一段话说清楚的事，不用列表。

**但是，出现 3 个以上并列项时必须用列表**（第一/第二/第三、一是/二是/三是、方式一/方式二/方式三）。用"第一...第二...第三...第四..."的段落形式写多个并列项，读起来既累又有浓重的 AI 味。

判断顺序（从上往下，命中即停）：
1. 并列项能用"，"或"、"连成通顺的一句话（如"支持 A、B、C 三种格式"）→ 段落
2. 并列项 ≥ 3 个 → 列表
3. 并列项只有 2 个，且每项不超过一句话 → 段落

### 金句与情绪表达（强制）

用 `## ending` 作为结尾，短句换行制造节奏，用具体场景代替抽象道理，可以用排比。

金句不再限定死板的位置，改为**触发管制 + 预算管制**：命中触发条件才允许写，全篇有总量上限。

**触发条件白名单**（只有命中才允许写金句，触发条件是"允许"，不是"任务"）：

- **开头前 3 段**：最多 1 处，用一句有态度的话立观点，立完马上回到书面语
- **`## ending`**：金句的主场，可以用 1-2 句，用【xxx】框起来
- **章节收尾判断句**：一个机制完整拆解完，用一句有态度的话收章（如"skill 不是收藏品，贵精不贵多"）
- **反直觉发现落地点**：证据刚展示完、读者正处在"原来如此"的情绪顶点，顺势给一句
- **取舍讨论之后**：设计代价讲清楚了，可以表个态

**硬禁区**（任何情况下保持科学表达，不安排金句）：

- 证据展示过程中（代码前后的"在做什么/意味着什么"两句）
- 参数、定义、步骤列表、表格的前后一句
- 技术机制的因果推理链条中间

**密度预算**：

- 全篇金句总量 2-4 处（含开头和 ending）
- 正文章节（`## 01` 到最后一个编号章节）最多 2 处，且不允许出现在相邻章节——连续两章都收金句，情绪就通胀了
- 每处 1-2 句，写完立刻回到书面语

**写法约束**：

- 金句必须从本文的具体场景里长出来，带着文章里出现过的细节
- 正文金句必须紧跟在证据或完整论证之后，情绪要有事实垫底。禁止无证据的抒情——这是它和 AI 味套话的分界线
- 禁止套格言公式（"X 是一面镜子""X 是 Y 时代的货币""X 是 Z 的语言"）——那是表演，不是表达。方向可以往这些靠近：工作的意义不只是赚钱、技术是为了让生活更美好

拿不准某处该不该用金句时，保守处理：只用开头和 ending 两个位置。读 `human-tone.md` 是为了找语感，具体使用范围见该文件开头的说明。

## 去 AI 味

读者看二哥是因为有判断、有喜好、有人情味。

**1. 禁止模板化结构**。

- 不要"说白了/本质上/核心在于"开头下定义
- 不要每段都用相同句式起手（连续出现"核心是..."、"关键是..."、"本质是..."属于典型 AI 味）
- 不要每段都用"这个设计的好处是..."、"这种方式的优势在于..."做过渡
- 不要每道题/每个章节都走"概念→解释→例子→总结"的固定模板，打乱顺序，有的先讲故事再给结论，有的直接上代码再解释为什么
- 不要标题后的废话垫场：标题下面先来一句重复标题意思的话再开始正文，直接进入内容
- 不要签到式过渡："让我们深入探讨""接下来看看""话不多说"，直接说事，不预告要说事
- 不要机械的"**加粗开头**：后接解释"式列表。列表项该加粗时加粗，但整篇所有列表都是同一个模子就是 AI 味

**2. 禁用 AI 味词汇和句式**：完整清单在 `references/human-tone.md`「AI 味禁用词汇」「AI 味禁用句式」两节，L1 自检 grep 词汇清单，L2 逐段扫句式清单。写作时的高频雷区：总结性套话（值得注意的是/综上所述）、互联网黑话（赋能/闭环/链路）、否定式排比（不仅……而且……）、格言公式（"X 是一面镜子"）、同义词轮换（同一个主语换着叫，该叫什么就一直叫什么）。

**3. 禁止绕弯凑篇幅**。一个技术点讲清楚就往下走，不要用"换句话说""也就是说""简单来说"再复述一遍。段落开头不铺垫，直接给信息。

**4. 禁止编造虚假案例凑字数**。不知道的去调查（步骤 1.5），调查不到主动问。

**5. 句式节奏**：长短句交替，不要连续同结构句（别连续三句"xxx是xxx"判断句）。适当用反问、设问调节节奏，但不要每段都加。段落结尾多样化，不要每段都总结句收尾。一个段落尽量不出现超过 3 个句号。

**6. 聚集判断，不要误伤**：单个信号不算问题——一个"然而"、一处排比、一个设问，人也会用。判断 AI 味看的是聚集密度：排比三连 + 格言公式 + 万能展望结尾同时出现才是供词。自检改稿时只清掉真正成片的模式，不要见一个杀一个，把文章扫得过度无菌反而不像人写的。

## 写出人味（正向技巧）

「去 AI 味」管的是别写什么，这一节管主动写什么。人味不是文字技巧，是文字背后站着一个付出过注意力的具体的人。

### 基础四条

- **具体到难以编造的细节**：真实的报错信息、真实的耗时数据、某次实测里的具体现象。这些细节来自步骤 1.5 的调查，编不出来，所以才可信
- **没有解决的矛盾心情**：评测类文章尤其要敢写"这工具大体不错，但有一处始终不顺手"。敢承认产品某处好某处烂，比通篇赞或通篇贬更像人。注意：矛盾心情用书面语表达，不靠口语腔
- **长短不一的呼吸**：真人的句子忽长忽短，AI 趋向均匀的中等长度。这是「句式节奏」那条的正向说法
- **用词有理由**：每个判断都能说出依据——是实测出来的、是源码里看到的、还是官方文档写的。说不出依据的判断不要下

### 命名技巧（工具箱，一篇取用 2-3 个即可）

6 个技巧的完整说明在 `references/human-tone.md`「命名技巧」一节，撰写前读：扣主线句、逐一展示（升番）、回环呼应、知识"顺手掏出来"、层层剥开、对立面先行。L3 自检要求至少用上 2 种。

## 特色元素

### 简历包装环节

文章涉及实战项目/GitHub 仓库时加一段：

项目名称
项目简介
技术栈
核心职责（5 条，公式：用了什么技术栈，解决了什么问题、实现了什么业务、有哪些量化数据，不能出现自定义类名）

### 截图占位符（强制）

每个章节（包括二级标题、三级标题、四级标题）至少 1 个占位符，超过 1000 字的章节安排 3 个占位符（前中后），格式：

```
【截图：<名称>；风格：<风格>；截图目标：<证明什么>；关键词：<关键词1>、<关键词2>、<关键词3>】
```

**风格只能选以下 6 种之一**，不能自造：

| 风格 | 适用场景 |
|------|----------|
| `whiteboard` | 默认白板架构图，适合系统架构、模块关系、数据流 |
| `skill-card` | 技能卡片、目录树、少文字的结构展示 |
| `data-board` | 统计看板、柱状图、热力表、数据对比 |
| `three-layer` | 三层结构、金字塔、台阶、层级关系 |
| `swimlane` | 泳道流程、多人协作、并行时序 |
| `checklist-card` | 注意事项、准备清单、避坑卡片 |

示例：

```
【截图：Skill 的结构；风格：skill-card；截图目标：展示 Skill 目录里 SKILL.md 必需，references/scripts/assets 可选，并且按需加载；关键词：SKILL.md、references、scripts、assets、按需加载】
```

## 工作流程

### 步骤 1：读素材 + 选题质检

精读 `./sucai.md`，提取关键信息、数据、观点、截图。素材中的截图可以直接搬进正文，减少改稿成本。

**选题质检（读完素材后立即执行）**：

用 IKR 三维度评估素材是否撑得起一篇好文章：

- **I (Insight)** 有没有一个读者不知道的信息差？（源码发现、实测数据、反直觉事实、从未有人做过的横向对比）
- **K (Knowledge)** 读完能带走什么？（可执行的操作、可复用的认知框架）
- **R (Resonance)** 能戳中什么情绪？（"原来如此"的恍然、"我也遇到过"的共鸣、"竟然可以这样"的惊叹）

三项占两项以上才动笔。只占一项时主动用 `AskUserQuestion` 问用户："素材在 X 方面偏弱，是否有补充信息？或者换一个切入角度？"

**推荐池质检（题材硬检查，IKR 之后立即执行）**：

2026 年 7 月后台数据结论（9 篇样本）：公众号推荐由题材归类决定，与互动指标（打开率、完读率、点赞、分享）无关。选题先归池：

- **AI 编程常青题材**（Agent 原理、Claude Code 机制、上下文工程、MCP、RAG、Multi-Agent——什么时候刷到都成立的问题）→ 垂直推荐池（实测推荐占比 37%～42%，长尾一个月），主力选题
- **时效新闻**（产品发布、公司动态、招聘绩效、行业榜单）→ 基本无推荐（实测 0.3%～9%，三天归零），只服务老粉点赞和搜一搜热词，不承担涨粉任务，压低成本写或建议用户改发星球
- **名人×科技破圈题材** → 泛流量池（阅读高但关注转化全场最差），每月至多一篇

素材落在后两池时，主动提醒用户该选题进不了垂直推荐池，确认是否继续。

### 步骤 1.5：调查真实细节（强制）

项目相关内容必须先调查再写，不能凭通用知识猜。编出来的细节一眼假，浪费时间。调查不到的用 `AskUserQuestion` 问用户，不要编造。

#### AI 角色边界

文章以二哥第一人称写作，但你是代笔，不是二哥本人。

**AI 负责做的**（放心生成）：
- 补充技术背景知识（框架原理、API 文档、源码解读）
- 找证据和佐证（官方数据、GitHub 星标、基准测试）
- 按确定的角度扩写段落内容
- 梳理逻辑结构、调整章节顺序
- 调查公开信息（仓库、文档、博客）

**必须问用户的**（AI 编了必假）：
- 第一人称经历（"我实测了三天""我一直在用 X"）
- 主观偏好和立场（"我觉得 A 比 B 好"）
- 私人情绪反应（"当时我就愣住了"）
- 未公开的内部信息

判断标准：如果一句话去掉"我"就能成立（纯技术事实），AI 可以写；如果去掉"我"就没意义了（主观体验），必须问用户或从素材中找。

#### 实测细节采集（产品评测类强制，其他类型按需）

高赞评测的核心燃料是真实体验细节，这些编不出来。素材（sucai.md 和截图）里找不到以下 4 项时，动笔前必须用 `AskUserQuestion` 向用户采集：

1. **实际花费/耗时**：花了多少钱、跑了多久（"Pro 一共花了 10 块左右""前后大概 15 分钟"）
2. **卡顿或意外**：过程中哪里不顺、哪里出乎意料（"修改文件明显卡顿，我是强制 ctrl+c 中断的""竟然跳过了 API key 接入，我也是很不解"）
3. **第一反应**：看到结果时的即时感受（"我个人的第一感觉还是非常喜欢的"）
4. **最终态度**：用完之后的定位——主力、备选还是弃用（"我目前的心态就是拿它作为一个备选选项"）

采集不到的项，对应的体验段落不写，宁缺毋假。深度拆解类对应采集的是"调查投入清单"：实际翻了哪些播客、论文、源码、文档，用于开场三件套的调查投入声明。
