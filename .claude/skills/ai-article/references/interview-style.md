# 面试对话类专属规范

> 仅在步骤 3 选择「面试对话类」时读取本文件。高赞正例全文见 `anshui-yin-mianshi.md`。

## 正文结构（覆盖通用规则）

`## content` 包裹所有面试题 → 每题 `### 01、问题`、`### 02、问题`。

**对话体框架**（高赞正例 `anshui-yin-mianshi.md` 的写法）：

- 面试官老王用对话体出场："老王问：""老王追问：""老王紧跟着追问："
- 场景描写用于节奏调剂："老王点点头""老王若有所思""老王拍了一下桌子"——全篇 3-5 处，不要每题都有动作，动作要和回答质量呼应（答得好才点头拍桌）
- 回答用"我说：""我回答："引出，或直接写内容，两种交替使用，避免每段都是同一个起手
- 长追问可以用四级标题 `#### 追问的问题内容` 承载，短追问用对话体，一篇文章两种混用

**对话体的边界（强制）**：对话框架是活的，回答内容是严谨的。老王的言行、"我"的临场反应走体验叙事轨；回答里的技术内容走技术论述轨——术语准确、数据具体、有出处。类比（"LSB 就像铅笔字，橡皮一擦就没了"）出现在准确解释之后，作为辅助理解，不替代解释。

示例：

```markdown
### 03、MCP 的 JSON-RPC 通信协议是怎么工作的

老王问："MCP 底层的通信协议了解吗？"

我说："JSON-RPC 2.0，很简洁，只有三种消息类型。第一种是 Request，有 id 字段，需要对方响应……"

老王追问："请求和响应的配对怎么实现的？"

"用一个 ConcurrentHashMap，key 是自增的请求 id，value 是 CompletableFuture……"
```

## 推荐池导向硬规则（强制，2026-07 后台数据验证）

背景：推荐流生客平均停留 42 秒、完读率 8%～12%，去留和转化全部发生在前几屏。最佳模板是“技术Leader惊了”篇——推荐 4,674 之外拿到全场最高的关注转化（0.42%）、点赞率（1.23%）和收藏率（2.37%），靠的是下面四条：

1. **第一屏续演标题对话**。生客是被标题里的面试冲突勾进来的，正文第一屏必须把这场对话演下去并兑现悬念，再展开技术内容。禁止先铺新闻背景、先讲概念再入场。
2. **前 600 字完成一次完整交付**：对话演完 + 第一个硬核答案给完，让 42 秒就离开的读者也带走了东西。
3. **可收藏干货负载**：中段安排一个读者想存下来的单元（面试官追问清单、答案卡、技能对照表），它同时是老粉的可转发单元。干货负载旁边紧跟系列感关注钩子（如“AI 面试连环拷问第 N 篇，关注看全系列”），钩子必须出现在前 600 字到中段之间，不放文末。
4. **自家项目（PaiCLI 等）推广 ≤ 一屏，且不在最后一屏**；简历包装的核心职责最多 3 条，其余引导去 GitHub。同一项目不连续两篇推。

**结尾三件套**（替代通用鸡汤）：与本篇话题绑定的表态句（敢站队）+ 一个具体的人或案例回扣 + 互动引导（“你怎么看，评论区聊聊”）。“AI 时代机会与挑战并存、加油吧”这类放在任何文章后面都成立的段落，一律删。

## 技术细节尺度（面试类强制）

**讲原理和思路，禁止出现项目自定义类名。**

- **禁止**：堆砌类名（如 `McpSchemaSanitizer`、`NotificationRouter`、`BrowserGuard`），面试时不可能记这些
- **保留**：Java 标准库 / 业界公认的技术名词可以用（如 ConcurrentHashMap、CompletableFuture、JSON-RPC、SSE、CDP、MCP、Function Calling），这些大家都认识
- 重点讲**为什么这样设计**、**解决了什么问题**，而不是**用了哪个自定义类**
